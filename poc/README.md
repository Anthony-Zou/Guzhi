# 故知 · KG-First 匹配引擎 POC

一个能跑的概念验证：用 Knowledge Graph 做确定性、可解释的人物匹配，
AI 只在最后一步做"故事推演"。

- **架构**：strict hexagonal（端口与适配器）
- **开发方式**：TDD（测试先行）
- **状态**：49 个测试全绿，30 人合成数据集上 F1 = 99.2%

```bash
python3 guzhi_poc.py        # 跑完整演示（命令行）
python3 -m pytest           # 跑全部测试（49 个）
python3 synthetic/run_synthetic.py   # 重新生成合成数据 + 验证
python3 viz/build_viz.py    # 生成可视化 -> viz/guzhi_viz.html（双击打开）
```

## 可视化

`viz/guzhi_viz.html` 是一个自包含的单页（无需服务器、无外部依赖）：
30 个合成人物画成力导向图，节点 = 人、连线 = 匹配。
- 节点颜色 = 涉及的最深语义簇等级（L3 暖橙 / L2 青苔 / L1 雾蓝）
- 节点大小 = 涉及的簇数量
- 实线 = 正确匹配（TP），红虚线 = 误配（FP），灰点线 = 算法漏掉（FN）
- 点节点 → 看 ta 的全部 KG 边；点连线 → 看共享簇、信号明细、故事种子

`viz/build_viz.py` 复用 POC 的匹配引擎导出数据，`viz/template.html` 是模板。

## 它做什么

输入：一组人，每个人是一张知识图谱子图（`(人)-[关系]->(实体)` 的边集合）。
输出：谁和谁匹配、为什么匹配（可解释）、以及一颗能交给 AI 的"故事种子"。

匹配**不靠 AI**。匹配是纯图运算：比较两个人各自指向的语义簇是否重叠。
AI 只在 `Narrator` 端口背后，负责拿故事种子写一段相遇对话。

## 算法核心：匹配要看"深度"，不只看"强度"

每个语义簇分三个深度等级：

| 等级 | 含义 | 单簇命中是否够 |
|---|---|---|
| L3 存在性议题 | 人最深层的东西（去留之惑、意义之问） | 够，门槛 0.07 |
| L2 价值观 | 怎么看世界（反效率主义、形式之美） | 够，但要 strength 乘积达标 |
| L1 偏好品味 | 喜好、风格（电影品味、户外行动） | 不够，必须叠加另一个簇 |

三道防硬凑闸：
1. **零共簇闸** — 没有任何共享簇 → 无匹配（这是排除"和谁都不搭"的人的机制）
2. **L1 单簇闸** — 只共享一个 L1 偏好簇 → 不够
3. **分级门槛** — 含 L3 簇用 0.07，否则 0.15

> 这套引擎可迁移：换语义簇定义和等级，就是 HR matching / co-founder matching。
> 两个候选人"都用 React"（L1）不构成团队契合；
> "都信任异步沟通、都受不了微观管理"（L2-L3）才是。

## 验证：为什么相信这个 99%

**8 人手工数据集不可靠**：样本太小（一对误差 = 5%），真值表是设计者
主观标的，命中率只有 50%。但这不能说明算法差 —— 它说明数据集差。

**30 人合成数据集才是真验收**：
- 人物由生成器从"特质库"组合而成
- 真值表是**生成规则的机械产物**（共享 ≥1 个 L3 簇，或 ≥2 个任意簇 → 该匹配），
  不是任何人的主观判断
- 关键防作弊：生成器判定匹配用**简单集合计数**，匹配算法用**带 strength /
  归一化 / 深度加权的打分** —— 两者逻辑不同。算法能复现生成器的意图，
  才说明算法的复杂打分确实在做对的事

结果：435 对里，Recall 100%、Precision 98.4%、F1 99.2%。

## 开发过程中抓到并修复的两个真 bug

都是靠"30 人合成数据 + 机械真值表"暴露出来的，手算和 8 人数据集都没发现：

**Bug 1 — 多簇贡献被 `max` 吞掉**
同一信号类别下多个簇命中（如两个 feeling_resonance 簇），代码用
`max` 只取了较大的，第二个簇白共享了。导致"共享两个 L3 簇"反而和
"共享一个"同分。修复：改成"概率或"式累加 `1-(1-a)(1-b)`。
（回归测试：`tests/test_matching_bug_multicluster.py`）

**Bug 2 — L3 门槛标定错误**
单个 L3 簇命中的典型分数是 0.07–0.13，原门槛 0.10 正好切在分布中间，
漏掉一半。门槛 0.10 当初是拍脑袋定的。修复：依据合成数据的实际分数
分布，降到 0.07。
（回归测试：`tests/test_matching_bug_l3_threshold.py`）

修完这两个，30 人 F1 从 92.4% → 99.2%。

## 这个 POC 还没回答的问题

- 真值表（哪怕是机械的）仍建立在"共享语义簇 = 该匹配"这个假设上。
  这个假设在真实世界成立吗？只能靠真实的人验证。
- 合成人物的边是干净的、簇标注是完美的。真实的人会模糊、会矛盾、
  会标注不全。
- 下一步：找 3–5 个真实的人，写真实自述、做真实标注、跑真实匹配。
  哪怕只有 10 对，也比 435 对合成数据更有说服力。
